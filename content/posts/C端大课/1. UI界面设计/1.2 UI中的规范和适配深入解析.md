## 1. 基本概念常识
>UI设计师的作品应用与电子屏幕上，设计师要对创作的载体有专业的认识
>显示器 - 笔记本 - 平板 - 手机
### 1.1. 像素
- **物理像素(px)**：屏幕显示器的**最小显示单位** → **硬件像素** (硬件==出厂自带，不可更改==)
	- eg.**像素风**(早年)：计算==硬件和性能局限==所使用数码绘图效果，使用“大矩形颗粒”完成图像的表达。 → 不同设备像素大小不统一![](https://int-1330323257.cos.ap-guangzhou.myqcloud.com//obsidian/pic/202602080159370.png)
		- **次/子像素渲染技术**(sub-pixel rendering)：在像素边缘增加==过渡色块==，使边缘圆润平滑![](https://int-1330323257.cos.ap-guangzhou.myqcloud.com//obsidian/pic/202602080157300.png)
### 1.2. 设计尺寸 ≠ 硬件尺寸
- **位图设计软件** → photoshop → 画布**有长宽单位**：px (==像素单位==) → 最后**屏幕**展现的==物理单位==
- **矢量设计软件** → figma → 画布**无长宽单位** (默认==矢量单位==pt/dp)——仅仅UI设计/程序员用的==逻辑单位==
	
	- VS位图：位图记录**全部像素信息**，系统直接**显示显示**什么
- 通过单位对比我们发现，设计尺寸(逻辑单位)，≠ 实际的硬件尺寸![](https://int-1330323257.cos.ap-guangzhou.myqcloud.com//obsidian/pic/202602081043358.png) ![](https://int-1330323257.cos.ap-guangzhou.myqcloud.com//obsidian/pic/202602081045919.png)
- **设计尺寸 → 硬件尺寸**(矢量 → 位图)：矢量记录的是图形**数学数据**，系统在**解析后渲染**后，在屏幕中显示出对应的**位图**![](https://int-1330323257.cos.ap-guangzhou.myqcloud.com//obsidian/pic/202602080217816.png)
### 1.3. 位图 VS 矢量图
- 位图：通过逐个记录**像素点色彩**，来保存图片的图像文件格式
	- 保存格式：
		- **未压缩** → ==完整记录==原图的数据，保存了多少记录多少还原多少 → 如==PNG、BMP== → 所以通常文件体积较==大==。
		- **压缩** → 是用一定==算法记录==图像数据 → ==JPG、WEBP== → 系统再对这些数据进行==解析显示==，所以通常文件较==小==。
			- 注：未压缩格式也可有“压缩”，例如使用==特定工具压缩PNG==，但这个压缩结果仅仅是通过减少图像本身数据的形式（==降低画质==）来实现的，并不是更改格式本身。
	- 优势：
		- **真实感强**：能精确记录照片、复杂纹理、渐变、阴影等细节。
		- **渲染简单**：显示时只需逐像素拷贝到屏幕，不需要复杂的几何计算。
	- 劣势：
		- **⭐⭐⭐不可无限缩放**：放大时会失真，无法保持清晰线条。
			- **像素数量固定，信息有限**。放大时只能靠插值“填补”，而插值无法创造真实细节，因此图像会失真。
		- **文件体积大**：高分辨率图像需要存储大量像素，文件容易很大。
		- **分辨率依赖性强**：显示效果完全依赖原始分辨率，跨设备时可能不够清晰。
- 矢量图：用**数学方式**(==点坐标、线方程、变换矩阵--缩放、旋转、平移==)**记录**(==文本或二进制方式==)图像形状、比例，来保存图片的图像文件格式
	- 保存格式： SVG (网页与跨平台矢量图)、EPS (印刷与出版)、PDF (文档)、AI (Al原生文件)、DWG (CAD 工程设计)
	- 优势：
		- **⭐⭐⭐无限缩放**：因为是数学定义，不依赖固定像素。
		- **文件小**：只需存储公式和参数，而不是每个像素。
		- **可编辑性强**：修改控制点或参数即可改变形状。
	- 劣势：
		- **不适合复杂、真实感强的图像**：难以表现照片那样的复杂纹理、渐变、阴影和细节
		- **渲染性能问题**：渲染时需要大量计算，可能比位图更耗费 CPU/GPU。
			- **渲染**本质：需要在显示时**实时计算几何公式并转换为像素**，而不像位图那样直接“**逐像素拷贝**”
		- **文件大小的反差**：矢量去模拟复杂图像（比如照片），文件会**比位图更大**，而且**效率低**。

### 1.4. 像素密度(PPI) VS 物理分辨率(PX²) VS 打印密度(DPI)
- **物理分辨率(px²=px乘px)** → 屏幕**总像素点**的个数 → 决定==细节上限==，容纳内容多少![](https://int-1330323257.cos.ap-guangzhou.myqcloud.com//obsidian/pic/202602080240877.png)
- **像素密度(PPI=px/inch)** →**每英寸屏幕**上能**显示**多少个**像素**→ 决定==清晰程度==，有没有颗粒感
- **点密度(DPI=Dots/inch)** →**打印机或扫描仪**在**一英寸内**能**打印/扫描**多少个**点** → 决定==打印精度==
	- 特殊：在平面设计和PS中，“分辨率” 指的是 **像素密度/点密度**(PPI/DPI), 像素密度决定了==印刷时的清晰度==
### 1.5. PT (DP/SP) 单位的引入
#### 1.5.1. 背景问题
1. **跨设备一致性**：不同设备的 **PPI**（像素密度）**差异很大**，直接用 px 会导致**同样的控件**在**不同设备**上**大小不一**。
	- 低PPI屏幕上，每个像素的物理尺寸更大 → 100px 占据的物理空间更大。
	- 高PPI屏幕上，每个像素的物理尺寸更小 → 100px 占据的物理空间更小。
2. **用户体验**
	- SP → 针对字体 → 允许用户调整系统字体大小 → 避免文字过小或过大，保证阅读舒适
3. **开发效率**
	- 开发者不必为每种屏幕**单独适配 px 尺寸**
	- 使用 DP/PT/SP 可以**自动适配**不同分辨率和屏幕尺寸
4. **设计与实现统一**
	- **设计师**用 pt/dp 定义界面元素，**开发者**实现时**保持一致**，减少**沟通成本**。

#### 1.5.2. 矢量单位的引入
- **PT(Point)** → **iOS** 的逻辑单位 →**1pt=1px**(163ppi--基准屏) → **1pt= 2/3px**(高密度屏--Retina 2x/3x屏)
    - 二倍屏(2x)：ppi=160 x 2 ，三倍屏(3x)：ppi=160 x 3
    - 设计师和开发者用 pt 来定义按钮、文字大小，而不用关心具体的物理像素。
	- ==人眼能分辨的像素点极限在300PPI左右==(视网膜屏一一Retina的定义)
- **DP(Density-independent Pixel)** → **Android** 的逻辑单位 →**1pt=1px**(163ppi--基准屏) → **1pt= 2/3px**
    - 在高密度屏幕上，系统自动缩放 dp → px，保持元素大小一致。
- **SP(Scale-independent Pixel)** → **Android 的字体**单位
	- 允许用户调整系统字体大小
    - 保证文字在不同设备和用户偏好下都能保持可读性。
#### 1.5.3. 逻辑尺寸与像素尺寸的转换(基于缩放因子)
- 缩放因子的**概念**(scale factor)： **逻辑单位 pt 映射到多少个物理像素 px**。
- 缩放因子的**计算**：==缩放因子 = ppi / 160== 
- 逻辑尺寸 → 物理尺寸：
- **系统层面**
	- 在 iOS 中，每个设备都有一个 UIScreen **.scale** 属性，里面的**scale就是缩放因子**：
	    - **普通屏幕**：**scale=1.0** → 1pt = 1px  → 每个==逻辑点==(pt²)由 **1×1=1** 个==物理像素==表示
	    - **Retina 2x 屏幕**：**scale=2.0** → 1pt = 2px → 每个==逻辑点==(pt²)由 **2×2=4** 个==物理像素==表示
	    - **Retina 3x 屏幕**：**scale=3.0** → 1pt = 3px → 每个逻==辑点==(pt²)由 **3×3=9** 个==物理像素==表示。
	- 开发者可以通过代码直接读取这个值来判断设备屏幕类型
	- 系统自动将**逻辑尺寸转换显示为物理尺寸** 
- **资源层面**
	- ⭐⭐⭐在 iOS 开发中，==图片资源需要提供不同版本==：
		- 普通屏幕：@1x
		- Retina 2x 屏幕：@2x
		- Retina 3x 屏幕：@3x
	- 系统会根据设备的**缩放因子**自动选择合适的资源。
#### 1.5.4. 矢量图渲染的基本逻辑
- 系统中记录的图形数据，需要经过一定的**计算和渲染**最终呈现到屏幕内。
	- 4K的图片放在1080P的电脑中显示，系统会把图片渲染成1080p（你需要的大小）进行显示，而不是只能显示它的四分之一。
	- 渲染逻辑图![](https://int-1330323257.cos.ap-guangzhou.myqcloud.com//obsidian/pic/202602080243992.png)
- 换算依据：**矢量大小×缩放因子**(ppi / 160) ，这个概念知道有这么一回事就可以，实际过程不会直接换算。
	- 定义一个图形是100，那么系统会根据设备的类型自动换算成100px、200px、300px进行显示，不需要我们手动去编写每台设备具体的数值。![](https://int-1330323257.cos.ap-guangzhou.myqcloud.com//obsidian/pic/202602080406207.png)
#### 1.5.5. 矢量画布大小的确定
**本质上取决于你设计的目标设备**。
- 不同屏幕有不同的分辨率(px²)和缩放因子(1x/2x/3x)，需要将其替换为**逻辑单位**，不是直接用物理像素(px)
- 实际设置的画布的大小，小要于手机屏幕分辨率![](https://int-1330323257.cos.ap-guangzhou.myqcloud.com//obsidian/pic/202602080411952.png)![](https://int-1330323257.cos.ap-guangzhou.myqcloud.com//obsidian/pic/202602080415588.png)

**figma中如何确定画布大小** (物理分辨率 ÷ 缩放因子 )
- iPhone 14 Pro
	- 物理分辨率：1290×2796 px  
	- 缩放因子：3x(=ppi/160)
	- 逻辑分辨率：430×932 pt (1290/3、2796/3)
	- 在Figma中，你只需用 430×932 的画布，系统会自动映射到 1290×2796 的物理像素。
	
**用PS创建UI画布**：要使用像素单位，
- 因为PS是一个位图软件，==如果使用1x画布就会导致无法导出更高清的2x、3x图片==。
- 应用PS做界面的场景往往是重运营、H5相关的页面
	- 完整的**3x**画布是一个**非常大**的画布，会导致软件的操作非常**卡顿**。
	- 为了减轻系统硬件压力，我们会在PS中创建**2x**的画布，可以**满足对应的切图和显示要求**（不会糊）![](https://int-1330323257.cos.ap-guangzhou.myqcloud.com//obsidian/pic/202602080422815.png)

#### 1.5.6. 拓展知识点(笔记本电脑)
**电脑端**的显示器**同样会使用倍率换算**(视觉的适配)
- **笔记本系统分辨率**设置
	- 因为**笔记本原本屏幕不够大**，而硬件分辨率又很高，当分辨率拉倒满，界面元素会变得非常小，所以只能采用**特殊方法**，来增大界面元素大小，从而实现视觉上的适配
	- mac电脑实现放大元素的方式
		- 使用==HiDPI== 技术，通过把1个“逻辑像素”映射到4个“物理像素”，对像素点合并，减小画布，实现清晰又易读的显示效果
			- **实际：硬件分辨率**：3840×2160（屏幕真实像素点）
			- **设置：系统分辨率**：1920×1080（**操作系统告诉软件的画布大小——欺骗的方法**）
			- **结果**：每个逻辑像素由 2×2 个物理像素渲染，视觉上更清晰但大小适中。
		- 劣势
			- 外接27寸屏，必须使用4k以上分辨率(物理像素足够多)，才能实现HiDPI的像素点合并
		- 表现场景
			- mackbook屏幕截图粘贴到软件里是2倍大(实际像素是两倍)
			- macbook中photoshop的画布要200%才能正常显示(ps是位图)
			- 接入4k屏，但是全屏网页没有两倍宽
	- wins电脑实现放大元素的方式
		- ==分辨率开到最大(硬件分辨率)==，通过增加 `缩放` 的比例到150%，来实现界面元素的放大![](https://int-1330323257.cos.ap-guangzhou.myqcloud.com//obsidian/pic/202602081106790.png)
		- 本质：通过**放大UI元素尺寸** 来实现可视分辨率的缩放
		- 优点：
			- windows画面分辨率 = 显示器分辨率/缩放值 → 相当可以**随意缩放画面分辨率** 
### 1.6. 总结
本质：
- 系统把 **pt 映射为多个 px**，是为了让 UI 元素在不同屏幕密度下保持**视觉尺寸一致**，同时利用高密度屏幕的优势提升清晰度。这就是 Retina 屏幕“看起来一样大，但更清晰”的本质。
- **逻辑像素**（PT/DP/SP）是**对物理像素**的**抽象**。
## 2. 移动端UI设计设计规范
### 2.1. 硬件规范
**硬件规范**：UI界面进行**显示**和**交互**的底层硬件规则
- **硬件分辨率≠设计分辨率**
- **交互逻辑**：手机端的触碰——**多点触碰** (电容屏)
	- 电容屏：可以多点触碰(放大缩小)
		- 应用于：智能手机、平板、消费电子。
	- 电阻屏：只能单点触碰(无法放大缩写)
		- 应用于：工业设备、医疗仪器、POS机等复杂环境
- **显示逻辑**：
	- 手机端——**更鲜艳**(LED)
		- LED：色彩更**鲜艳**，**对比度高**，更亮，节能，支持**区域控光**
		- LCD：色彩较**暗淡**，**对比度低**，**亮度**有限
	- 移动端有多种**像素密度**的**屏幕规格**：PPI=分辨率/屏幕尺寸
		- 165ppi → 320 × 480px
		- 165ppi → 848 × 1792px
		- 165ppi → 1644 × 3840px
	- **使用物理像素定义尺寸无法在设备间通用** 
		- → 开发环境下使用**矢量单位**(逻辑单位)设置元素尺寸(pt/dp/sp)
	- 手机系统会自动**还原**实际的**像素**数值
		- 像素尺寸 = 逻辑尺寸 × 缩放因子
		- 缩放因子 = 设备像素密度/160
		- 图示：![](https://int-1330323257.cos.ap-guangzhou.myqcloud.com//obsidian/pic/202602081131665.png)
### 2.2. 系统规范
系统规范：界面所在系统的**官方规范标准**
- **IOS系统规范**：UI设计主要以IOS规范为标准开展
	- IOS总体规范：[链接](https://developer.apple.com/cn/design/human-interface-guidelines/) 
- **安卓系统规范**：只有面向安卓的产品才优先考虑安卓规范
	- 要学会从**官方页面**中查找相关信息资料，不要从第三方获取，官方地址
	- HarmonyOS： [设计理念-通用设计基础 - 华为HarmonyOS开发者](https://developer.huawei.com/consumer/cn/doc/design-guides/design-concepts-0000001795698445)
- **官方建议**的设计**画布尺寸**：![](https://int-1330323257.cos.ap-guangzhou.myqcloud.com//obsidian/pic/202602081135168.png)
	- 一般选择创建偏大一点或者中间尺寸的画布尺寸(主流)
	- 在figma的**assets**中搜索**最新尺寸**
- ⭐⭐⭐**IOS界面设计基本结构**
	- IOS 18(18代)：状态栏status bar 54pt、顶部栏 44pt、标题栏 52pt、tab bar 49pt、home indicator 34pt 
	- IOS 26(26**年**)：状态栏status bar 62pt、顶部栏 44pt、标题栏 41pt、tab bar 62pt、home indicator 21pt![](https://int-1330323257.cos.ap-guangzhou.myqcloud.com//obsidian/pic/202602081208728.png)
		- 状态栏用==适配的尺寸== → 才能**适配灵动岛尺寸!!!**
		- home indicator**区域尺寸缩小**了 → 因为26以后，默认用户已经熟悉home indicator了，所以隐藏了
	- ⭐⭐⭐**安卓端**
		- **安卓端**的设备过多，所以头部和底部组件**没有标准样式**，可以从自己手机截图或使用Material官方**组件**替代做示意，**无法代表最终显示效果**。
		- **官方**有提供不少**组件**，但大多仅仅是作为基础可选服务，你**可以用**(便于开发)也**可以不用**，而布局形成了很多非官方的行业规范、套路。![](https://int-1330323257.cos.ap-guangzhou.myqcloud.com//obsidian/pic/202602081214820.png)
- **移动端的主要字体应用**
	- 字体选择
		- 界面设计中字体通常选择**面向系统**的字体，部分情况下**设计稿的字体仅是示意**，实际用户本地查看时无法完全一致（尤其安卓）。
		- **iOS**官方的字体主要是**针对英文**制定的，**中文**的移动端应用实际上只需要**牢记下方的属性定义数值**即可。
		- IOS：苹方、SF Pro
		- Android：思源黑体、HarmonyOS Sans、Mi Sans、Oppo Sans 
	- 中文字体大小：中文应用设计无需遵照英文的字体规范![](https://int-1330323257.cos.ap-guangzhou.myqcloud.com//obsidian/pic/202602081218993.png)
	- 文本对齐
		- 文字的行高会大于字体从而预留出上下的空隙
		- 排版过程以**文本的外边缘为准**而==不是字体轮廓==![](https://int-1330323257.cos.ap-guangzhou.myqcloud.com//obsidian/pic/202602081222372.png)
	- SF Pro 的字体设置
		- 注意：iOS使用纯英文字体 SF Pro的话，系统会根据字号大小自动切换 Display (字体不小于20)和 Text(字体小于20)两个细分字体。
			- SF Pro Text(正文)：字母**负空间**小
			- SF Pro Display(标题体)：字母**负空间**更大，不同字重都需要单独设置字母间距
			- 图示：![](https://int-1330323257.cos.ap-guangzhou.myqcloud.com//obsidian/pic/202602081226371.png)
	- 行间距：调整到适合阅读
	- 中英文混排：不需要单独选中英文设置，也不需要在中文应用中随便使用 SF Pro系列字体，**统一用中文字体**
	- 4的倍数
		- 系统规范建议使用4的倍数设置参数，但 **"建议型"** 规范需要根据场景判断是否遵守。
		- 有**很多场景无法使用4的倍数设置**，如自适应尺寸、字体大小、图标尺寸等等。设计师要**根据对场景的判断决定**应该使用什么参数。

## 3. UI中的布局和排版
- 扁平UI**界面设计本质**是：元素排布 + 元素设计 + 图标设计UI
- 界面有4个基本**构成要素**：几何、图片、图标、文字。
- **界面设计流程**：==先排版 → 后填充 → 做设计== 
	- 类似：绘画：整体人物轮廓 - 细节绘制 - 颜色填充
	- 图示：![](https://int-1330323257.cos.ap-guangzhou.myqcloud.com//obsidian/pic/202602081229447.png)
- **界面排版**：==即对尺寸和间距控制的过程==
- 整个移动端**元素的尺寸设置**，包含两种模式：：
	- **预设尺寸**：尺寸是==直接定义==的，根据规范、效果、个人喜好等，定死。(字间距，icon尺寸)
	- **适应尺寸**：尺寸是==根据下级内容==或==外部环境决定==的，可变。(卡片宽高、栏目高度、标签宽度等组合元素)
	- 图示：![](https://int-1330323257.cos.ap-guangzhou.myqcloud.com//obsidian/pic/202602081231746.png)
- **间距的定义**：定义元素的内间距和外间距
	- 以4的倍数为准，并结合亲密性原则，定义间距(==2/4/8/12/16/20)==![](https://int-1330323257.cos.ap-guangzhou.myqcloud.com//obsidian/pic/202602081232172.png)
- **间距和使用尺寸的结合，就是不同屏幕适配的基础**：![](https://int-1330323257.cos.ap-guangzhou.myqcloud.com//obsidian/pic/202602081233805.png)
- 减去间距后，**卡片平均大小有小数**
	- 小数可以用**近似整数**代替，因为在实际开发过程中，卡片大小为未知数x，所以程序员能看懂
		- 左卡片(**174.5→175**)，右卡片(**174.5→14**) 
- 界面排版遵循**从上到下**的制作排版顺序，是使用类似 **"拼积木"** 的方式逐步填充内容
- **"临摹是最好的练习"**
	- 不是1:1还原，而是==理解布局意图==，并二次实现
## 4. UI排版的训练
对元素大小的控制、间距的控制，就是U排版中前期最重要的练习部分，需要将这些操作的规则、习惯掌握到滚瓜烂熟，就像学写作文之前先要把字写对。
而认字、写字和措辞、造句水平没有直接的联系，在UI设计中，**布局本身的掌握也和色彩、用图、图标、动效等技能没有相关性**，要==单独进行训练==。界面做得差大概率也是布局本身的问题，加强其它技能不会有任何改善。
最好的训练方式，就是**只用原型图的模式绘制界面的基础框架**，在这个阶段不需要上色、填图、绘制图标。![](https://int-1330323257.cos.ap-guangzhou.myqcloud.com//obsidian/pic/202602081235497.png)
**同时，所有设计结果都需要在自己手机上查看结果。**
